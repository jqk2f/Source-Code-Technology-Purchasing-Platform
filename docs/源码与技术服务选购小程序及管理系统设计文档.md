适用范围：客户端小程序、管理后台 Web、服务端 API、数据库、文件与消息服务

## 1. 文档说明

### 1.1 建设目标

本设计文档用于指导“源码产品、软件成品、技术服务选购平台”的研发落地。系统面向个人运营场景，核心闭环为：

1. 管理员维护商品、服务、套餐、内容与配置。
2. 客户在小程序浏览、搜索、收藏、咨询、提交购买意向或定制需求。
3. 管理员跟进询单、报价、转订单。
4. 客户线下付款并上传凭证。
5. 管理员人工确认收款、维护交付资料、处理售后。

### 1.2 当前版本边界

当前版本不接入小程序在线支付、不做资金托管、不做第三方程序员入驻、不做竞标派单、不做钱包提现和分账。系统仅记录人工收款、交付和售后过程，并预留后续在线支付、自动发货、电子合同等扩展字段。

### 1.3 技术假设

| 项目 | 建议方案 |
| --- | --- |
| 客户端 | 微信小程序 |
| 管理后台 | Vue / React 管理端均可 |
| 服务端 | Java Spring Boot / Node.js NestJS / PHP Laravel 均可，本文接口按 REST 设计 |

| 消息 | 站内信为必做，微信订阅消息为可选 |
| 鉴权 | 客户端 JWT / Session Token，后台 JWT + RBAC |

## 3. 核心业务设计

### 3.1 商品购买流程

1. 客户浏览商品详情并提交购买意向。
2. 系统生成询单，状态为 `pending_follow`。
3. 管理员沟通后报价，询单状态变为 `quoted`。
4. 客户确认后，管理员将询单转为订单，订单状态为 `pending_payment`。
5. 客户上传付款凭证，订单状态变为 `payment_confirming`。
6. 管理员确认收款，若全额收款则订单状态变为 `pending_delivery`。
7. 管理员添加交付记录，订单状态变为 `pending_acceptance` 或 `completed`。
8. 客户确认交付后订单完成，后续可发起售后。

### 3.2 服务预约流程

1. 客户选择服务并填写问题说明、环境、截图和期望完成时间。
2. 系统生成服务类询单。
3. 管理员评估范围、报价并转订单。
4. 客户付款并上传凭证。
5. 管理员确认收款后，订单进入 `in_service`。
6. 服务完成后添加服务结果或交付说明，订单进入 `pending_acceptance`。
7. 客户确认后完成。

### 3.3 定制需求询价流程

1. 客户提交定制需求。
2. 管理员记录沟通内容、报价、周期和下次跟进时间。
3. 达成一致后转订单。
4. 若分阶段交付，当前版本通过多笔收款记录和多条交付记录表达，不做资金托管。

### 3.4 收款确认规则

| 场景 | 处理规则 |
| --- | --- |
| 客户上传凭证 | 创建收款记录，状态 `pending`，订单进入 `payment_confirming` |
| 管理员确认到账 | 收款记录状态 `confirmed`，累计实收金额 |
| 实收金额小于应收金额 | 订单付款状态 `partial_paid`，订单状态可保持 `partial_paid` 或继续待付款 |
| 实收金额大于等于应收金额 | 订单付款状态 `paid`，订单进入待交付或服务中 |
| 凭证无效 | 收款记录状态 `rejected`，记录驳回原因 |
| 收款作废 | 仅超级管理员或有权限角色可操作，写操作日志 |

### 3.5 订单状态机

| 状态码 | 状态名 | 进入条件 | 可流转到 |
| --- | --- | --- | --- |
| `pending_communication` | 待沟通 | 客户提交意向后生成订单，或询单转订单但未确认内容 | `pending_confirm`, `cancelled`, `closed` |
| `pending_confirm` | 待确认 | 管理员已沟通，等待客户确认报价或购买内容 | `pending_payment`, `cancelled`, `closed` |
| `pending_payment` | 待付款 | 订单金额已确认，等待客户付款 | `payment_confirming`, `partial_paid`, `cancelled` |
| `payment_confirming` | 待确认收款 | 客户已上传付款凭证 | `pending_delivery`, `in_service`, `partial_paid`, `pending_payment`, `cancelled` |
| `partial_paid` | 部分收款 | 已确认部分金额但未全额 | `payment_confirming`, `pending_delivery`, `in_service`, `cancelled` |
| `pending_delivery` | 待交付 | 商品类订单已完成收款 | `pending_acceptance`, `completed`, `after_sales` |
| `in_service` | 服务中 | 服务类订单已确认收款 | `pending_acceptance`, `after_sales`, `cancelled` |
| `pending_acceptance` | 待验收 | 已添加交付或服务结果 | `completed`, `after_sales` |
| `completed` | 已完成 | 客户确认或管理员确认完成 | `after_sales`, `closed` |
| `after_sales` | 售后中 | 存在进行中的售后工单 | `completed`, `closed` |
| `cancelled` | 已取消 | 客户或管理员取消 | `closed` |
| `closed` | 已关闭 | 超时、失效或无需继续处理 | 无 |

## 4. 数据库设计

### 4.1 设计约定

1. 数据库字符集：`utf8mb4`。
2. 表引擎：`InnoDB`。
3. 主键：除特殊说明外使用 `bigint unsigned` 自增主键。
4. 金额：使用 `decimal(10,2)`，单位为元。
5. 时间：使用 `datetime`，服务端统一保存标准时间。
6. 软删除：业务主表建议使用 `deleted_at`。
7. 通用审计字段：`created_at`、`updated_at`、`created_by`、`updated_by`。
8. 状态、类型字段使用英文枚举码，前端展示由字典接口或本地映射转换。

## 5. 接口规范

### 5.1 通用约定

#### 请求约定

| 项目 | 说明 |
| --- | --- |
| Base URL | `/api/v1` |
| Content-Type | `application/json; charset=utf-8` |
| 文件上传 | `multipart/form-data` |
| 客户端鉴权 | `Authorization: Bearer <customer_token>` |
| 后台鉴权 | `Authorization: Bearer <admin_token>` |
| 时间格式 | `YYYY-MM-DD HH:mm:ss` |
| 分页参数 | `page` 默认 1，`pageSize` 默认 20，最大 100 |

#### 通用响应

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "traceId": "202605211600001234"
}
```

#### 分页响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],
    "page": 1,
    "pageSize": 20,
    "total": 100
  }
}
```

#### 常用错误码

| 错误码 | 说明 |

## 6. 权限与安全设计

### 6.1 客户端数据权限

1. 客户只能查看自己的询单、订单、付款凭证、交付资料和售后记录。
2. 商品、服务、分类、标签、公开文章允许未登录访问。
3. 收藏、订单、交付、售后、消息必须登录。
4. 私有文件下载前必须校验业务归属，例如订单客户 ID 是否匹配。

### 6.2 后台 RBAC 权限

| 角色 | 权限范围 |
| --- | --- |
| 超级管理员 | 全部菜单与操作权限 |
| 运营 | 商品、服务、套餐、内容、统计，不含敏感收款确认 |
| 客服 | 客户、询单、订单跟进、售后回复，不可作废收款 |
| 交付人员 | 订单查看、交付资料维护、售后协助 |
| 只读人员 | 只读查看，不可编辑、删除、导出 |

### 6.3 敏感信息保护

1. 手机号、微信号、付款凭证、交付账号等敏感信息后台按权限脱敏展示。
2. 交付内容中如包含账号密码，建议字段级加密保存。
3. 私有文件使用短期签名 URL，不直接暴露永久地址。
4. 重要操作写入操作日志，包括收款确认、收款作废、交付资料修改、订单金额修改。
5. 文件上传限制格式、大小，并接入安全扫描或内容审核能力。

### 6.4 接口安全

1. 登录、上传、提交询单、上传凭证接口需要限流。
2. 后台登录连续失败应触发短时锁定或验证码。
3. 所有金额修改和状态流转均在服务端校验，不信任前端传入状态。
4. 所有列表查询需要分页，避免大查询。
5. 导出接口需要权限控制和操作日志。

### 7.3 私有文件下载

私有文件下载流程：

1. 客户或管理员请求下载接口。
2. 服务端校验登录态和业务权限。
3. 服务端生成短期有效签名 URL。
4. 记录下载日志。
5. 客户端使用签名 URL 下载。

## 10. 非功能设计

### 10.1 性能指标

| 场景 | 指标 |
| --- | --- |
| 小程序首页首屏 | 2 秒内返回核心数据 |
| 商品列表接口 | 500 毫秒内返回 |
| 商品详情接口 | 800 毫秒内返回 |
| 后台列表查询 | 1 秒内返回 |

### 10.2 可用性

1. 商品浏览、询单提交、订单查看、交付查看为核心链路，需要优先保障。

### 10.3 扩展性

1. `orders.payment_status`、`payment_records.payment_method` 已预留在线支付扩展。
2. `order_items.snapshot` 支持保存下单时商品快照，避免商品变更影响历史订单。
3. 商品、服务、询单、订单均使用类型字段支持新业务形态。
4. 配置项、分类、标签、付款方式、交付类型可字典化扩展。

### 12.1 核心测试用例

| 模块 | 测试点 |
| --- | --- |
| 登录 | 微信登录、Token 过期、手机号绑定 |

| 订单 | 询单转订单、金额修改、订单状态流转 |
| 收款 | 客户上传凭证、后台确认、驳回、部分收款、作废 |
| 交付 | 添加交付、客户查看、客户确认、私有权限校验 |
| 售后 | 创建售后、后台回复、解决、关闭 |
| 权限 | 不同后台角色菜单与操作权限 |
| 文件 | 公开文件访问、私有文件鉴权、格式和大小限制 |
| 消息 | 关键状态变更生成站内信 |

### 12.2 验收标准

1. 管理员可新增、编辑、上架、下架源码商品和技术服务。
2. 客户可浏览、搜索、筛选、收藏商品和服务。
3. 客户可提交购买意向、服务预约和定制需求。
4. 管理员可跟进询单、报价并转为订单。
5. 管理员可修改订单金额、确认订单、取消订单和完成订单。
6. 客户可上传付款凭证，管理员可确认、驳回、作废收款记录。
7. 管理员可添加交付资料，客户只能查看自己订单下的可见交付。
8. 客户可提交售后，管理员可处理和关闭售后。
9. 后台可按权限控制敏感操作，关键操作有日志。
10. 系统不依赖在线支付也能完成从询单到收款、交付、售后的业务记录。
